# 작업 핸드오프 — 2026-04-19

## 오늘 완료한 작업

### 1. Contextual Tab 자동 수집 추가 (menu-mapper)

HWP 2027의 contextual tab 중 **"객체 선택" 계열**(도형/표/차트/그림)에 이어 **"캐럿 컨텍스트" 계열**(메모/주석/머리말·꼬리말)도 자동 수집하도록 확장.

#### 변경 구조
- `src/menu-mapper.js`:
  - `_probeContextualTabByEntry()` 공통 헬퍼 추가 (diff 기반 새 탭 탐지, 리본 수집, 드롭다운 프로빙, 리본 내 닫기 버튼 자동 클릭 cleanup)
  - `_probeMemoTab()`, `_probeAnnotationTab()`, `_probeHeaderFooterTab()` 신규 프로버
  - Step 9.5 (메모), 9.6 (주석), 9.7 (머리말/꼬리말) `run()` 플로우 편입
  - 기존 Step 12 (`probeEditModes`) 및 `_probeEditMode`/`_probeMemoEditMode`/`_probeFootnoteEditMode`/`_probeEndnoteEditMode`/`_probeHeaderFooterEditMode` 전부 제거
  - 옵션 플래그: `probeMemoTab`, `probeAnnotationTab`, `probeHeaderFooterTab` (기본 true)
- `src/verify-edit-modes.js`: DEPRECATED 주석만 추가

#### 검증 결과 (`test-probe-caret-final.js`)
| probe | 결과 | 비고 |
|---|---|---|
| `_probeMemoTab` | ✅ pass — `map.tabs['메모']` 11개 리본 + 3 드롭다운 | `contextState: 'memo-caret'` |
| `_probeAnnotationTab` | ✅ pass — `map.tabs['주석']` 19개 리본 + 5 드롭다운 | `contextState: 'annotation-caret'`, "닫기 : ALT+X" 버튼 자동 클릭 |
| `_probeHeaderFooterTab` | ⚠️ 스킵 (알려진 한계) | 아래 "한계" 섹션 참조 |

#### 중요한 발견
- 입력 탭의 실제 accessKey는 **`D`** (과거 `_probeMemoEditMode`가 `Alt+I`를 썼는데 잘못된 키였음). 현재는 `_switchTab` 마우스 기반 전환 사용.
- 6개 HWP 프로세스가 동시에 실행될 수 있는 환경에서는 `win32.mouseClick` 직접 호출 대신 **`_safeClick`(`_ensureForeground` 선행)** 이 필수.

### 2. 스트레스 테스트 검증 파이프라인 V1/V2 구축

기존 `stress-runner`가 action 실행만 로깅하던 것에 더해, **"의도된 동작이 실제 일어났는지"** 를 before/after snapshot 비교로 검증하고 지속 개선 가능한 리포트를 생성.

#### 변경 구조

### 3. 스타일 드롭다운 항목의 짧은 이름 — 조사 결과(결론: 불가)
UIA의 모든 속성 (`name`, `automationId`, `acceleratorKey`, `localizedControlType`, `className`)이 짧은 이름을 담고 있지 않음. HWP 2027이 스타일 드롭다운 내부를 커스텀 렌더링하기 때문. 도형 드롭다운은 `name`에 짧은 이름이 그대로 있어 정상 작동.

## 알려진 한계 / TODO

### (A) 머리말/꼬리말 contextual tab 자동 진입 불가
`src/menu-mapper.js:_probeHeaderFooterTab()` 내부에 상세 주석으로 기록됨.
- 머리말 드롭다운은 `_probeDropdown`으로 열 수 있음 (아이템 3개: 위쪽/왼쪽-오른쪽/머리말-꼬리말 확인)
- 그러나 각 아이템이 **키보드·마우스 클릭·UIA `invoke()` 어느 수단에도 응답하지 않음**
- 10가지 이상 전략 시도 실패 (테스트 기록: `test-results/caret-diag*.log`)
- **후속 작업**: 머리말 드롭다운 아이템이 사용하는 내부 이벤트 구조 재조사 필요. 다른 진입 경로(예: `Ctrl+N, H` 조합 등) 발견 시 `_probeHeaderFooterTab`의 `enterFn` 교체.

### (B) `hasDropdown` 맵 오분류 (서식.취소선, 그림.밝기 등)
- stress fail에서 "dropdown open-only but state changed"로 감지됨
- 맵에는 `hasDropdown=true`인데 실제는 즉시 적용 토글
- **후속 작업**: 맵에서 `dropdown.classification === 'no-popup'`인 항목을 찾아 `map-to-cases`에서 `action` 카테고리로 재분류하는 규칙 추가

### (C) V3 검증 개선 — `isModified` 포화 문제
대부분의 서식 토글/표 편집 action이 unknown으로 남는 근본 이유. 제안 방향 (우선순위순):
1. **V3-A: 본문 text 길이 / 해시 비교** — snapshot에 `bodyTextHash` 추가. COM `InitScan/GetText`로 본문 전체 읽어 길이 또는 xxhash. 비용 약 30ms.
2. **V3-B: 선택 영역 CharShape/ParaShape 비교** — `_hwp.CharShape.Bold` 등 속성 diff로 서식 토글 직접 감지. 선택 있을 때만 유효.
3. **V3-C: 전체 Ctrl 목록 해시** — `HeadCtrl`부터 `Next()` 순회. 메모/각주/도형 삽입을 필드리스트 대신 ctrl 추가로 감지.
4. **V3-D: HAction.LastCommand 등 소프트 시그널** — HWP가 이런 API 노출하는지 조사 필요.

### (D) 스타일 드롭다운 짧은 이름 — 보류
UIA 단독으로는 획득 불가 확인. COM 기반 순서 매칭(지표 zipping) 또는 클릭-후-읽기 방식이 있으나 위험/비용 대비 효용 낮음. 짧은 이름 없이도 stress runner는 rect 좌표로 정상 동작.

### (E) 4건 fail의 재현 및 원인 확정
